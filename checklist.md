🚩 Prompt2Repo 产物最终质检手册 (Final Checklist)
一、 硬性准入检查 (Must-Have)
只要有一项不通过，直接判定为“不可验收”或“打回”。

[ ] Docker 一键启动：执行 docker compose up 是否能直接进入可用状态？

[ ] 环境隔离：是否完全不依赖宿主机本地安装的 Node.js, Python 或数据库？

[ ] Prompt 覆盖率：是否实现了 prompt.md 中要求的所有核心功能（含业务逻辑、UI 要求）？

[ ] 无偏离实现：产物是否紧扣题目主题，没有擅自修改核心业务逻辑或简化需求？

[ ] 运行稳定性：在不修改任何代码的情况下，项目是否能稳定运行且无崩溃？

二、 工程架构与目录规范 (Engineering)
评估要点：结构是否专业，是否具备 0-1 的完整形态。

2.1 标准目录结构
[ ] 层次分明：代码是否按功能分层（如 /src/components, /src/services, /src/api）？

[ ] 逻辑解耦：禁止在单一文件内堆叠大量逻辑（例如：禁止在 API 路由文件中直接写长达百行的业务逻辑，应提取至 Service 层）。

[ ] 配置文件：是否存在 .env.example？（严禁提交含有私钥的 .env）。

[ ] 产物清理：是否删除了 .git, .next, node_modules, .DS_Store，ide提供的临时产物 等冗余文件？

2.2 数据库与持久化
[ ] 自动初始化：数据库启动后是否自带基础测试数据（Seed Data）？

[ ] 连接规范：容器间访问是否使用了 Service Name（如 db:3306）而非 localhost？

三、 代码细节与专业度 (Quality)
评估要点：是否体现三年以上经验工程师的水准。

[ ] 错误处理：对后端 API 的异常是否做了 try-catch？前端是否能优雅展示错误（如 Toast）而非控制台报错？

[ ] 日志记录：关键业务路径是否有 Logger 输出以便排查问题？

[ ] 参数校验：对于前端输入或接口参数，是否做了基本的内容校验（如非空、格式校验）？

[ ] 代码整洁：是否清理了冗余的 console.log 和无意义的注释？

四、 UI/UX 与美观度 (Aesthetics)
仅针对全栈及纯前端题目，需符合“美观度交互式标注”标准。

[ ] 视觉分区：页面不同功能区域（导航、侧边栏、内容区）是否有明确的视觉区分（背景色、留白、边框）？

[ ] 响应式/对齐：页面元素是否对齐，间距（Padding/Margin）是否统一？

[ ] 交互反馈：点击按钮是否有 Loading 状态？Hover 元素是否有样式变化？

[ ] 素材一致性：使用的图标、字体、占位图是否符合页面主题？（禁止出现明显的风格违和）。

五、 交付文档规范 (Documentation)
评估要点：QA 人员能否“不读源码”就完成验证。

[ ] README 完整性：必须包含以下标准段落：

## How to Run (必须是 docker compose up)

## Services (列出访问地址，如 http://localhost:3000)

## Verification (简述如何通过 3-5 个步骤验证核心功能)

[ ] 自测截图：领题表中的“自测情况”是否包含 6 个维度的实际运行截图？

六、 废弃题目快速排查表
如发现以下情况，请立即停止标注并反馈给 POC：

黑盒 API：依赖无法 Mock 的第三方 SaaS 服务（如需真实 Key 的支付、AI 接口）。

硬件/系统限制：题目强制要求 Windows 系统或特定的外设硬件。

内容残缺：Prompt 引用了图片或文档但未在清单中提供。
